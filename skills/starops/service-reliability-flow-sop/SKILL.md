# 业务服务可靠性巡检 — SOP Skill

> 本仓 **SOP Skill** 入口：把 [人读版实践文档](https://sls.aliyun.com/doc/starops/practices/business-reliability-flow/) 的 SOP 固化为 Agent 可加载形态。
>
> 区分：本实践无 meta skill 样品（产物是业务可靠性报告，不是可执行业务 Skill）。

## SOP 概览

5 phase 端到端流程，章节顺序与 [人读版实践文档 步骤一～步骤五](https://sls.aliyun.com/doc/starops/practices/business-reliability-flow/) 一一对应：

1. 业务指标基线 — 核心业务指标的当前值 + 历史基线对比
2. 应用指标关联 — 从业务指标异常 → 关联应用层指标
3. 依赖拓扑分析 — 从异常应用 → 追踪上下游依赖，定位瓶颈
4. 告警事件聚合 — 聚合相关告警和变更事件
5. 综合可靠性报告 — 综合 Phase 1-4，输出可靠性评估

## 步骤 1：业务指标基线

**Agent 动作**：向 STAROps 发起业务指标基线查询。

**工具调用**：

```
@应用-<服务名> 对最近 <时间范围> 的核心业务指标做基线分析：QPS、成功率、P99 延迟，对比前 <基线天数> 同期基线，判断是否偏离正常范围
```

> 注意：切入点选核心业务链路的入口服务（如 checkout / frontend），而非底层依赖服务。prompt 必须显式指定服务名。

**输出要求**：
- 业务指标基线表（指标 / 当前值 / 基线值 / 偏差百分比 / 状态）
- 关键发现：哪些指标偏离基线、偏离幅度
- 周期性波动说明：是否存在预期内的波动（如白天高峰 / 夜间低谷）

**L0 声明**：本步骤只读，不发起任何变更。

## 步骤 2：应用指标关联

**Agent 动作**：向 STAROps 发起应用指标关联查询，从业务异常追溯到应用层。

**工具调用**：

```
@应用-<服务名> 基于业务指标基线分析结果，关联最近 <时间范围> 的应用层指标（错误率 / QPS / P99 延迟 / 实例数 / CPU / 内存），定位哪些应用指标异常可能影响业务
```

> 注意：本步骤依赖 Phase 1 的业务指标基线结果。确保在同一 thread 内执行。

**输出要求**：
- 应用指标关联表（指标 / 当前值 / 基线值 / 偏差 / 与业务指标的关联）
- 因果分析：哪些应用指标异常与业务指标偏离有因果关系
- 异常排序：按对业务影响程度排序

**L0 声明**：本步骤只读，不发起任何变更。

## 步骤 3：依赖拓扑分析

**Agent 动作**：向 STAROps 发起依赖拓扑查询，从异常应用追溯到上下游依赖。

**工具调用**：

```
@应用-<服务名> 分析当前服务的上下游依赖拓扑，标注最近 <时间范围> 有异常（错误率升高 / 延迟增大 / 连接超时）的依赖服务，定位瓶颈或故障点
```

> 注意：依赖拓扑依赖 OpenTelemetry trace 数据。如果 trace 采样率过低，拓扑可能不完整。

**输出要求**：
- 依赖拓扑图（上游 → 当前服务 → 下游依赖）
- 异常依赖标注表（依赖服务 / 关系 / 错误率 / 延迟 / 状态）
- 瓶颈定位结论：哪个依赖服务是当前瓶颈

**L0 声明**：本步骤只读，不发起任何变更。

## 步骤 4：告警事件聚合

**Agent 动作**：向 STAROps 发起告警和变更事件聚合查询。

**工具调用**：

```
@应用-<服务名> 聚合最近 <时间范围> 内与该服务及其依赖相关的所有告警和变更事件，按时间线排列，标注哪些可能与业务指标异常相关
```

> 注意：告警和变更事件来自不同数据源（告警系统 / 变更管理系统），AI 需要跨数据源聚合。

**输出要求**：
- 告警/事件时间线表（时间 / 类型 / 来源 / 内容 / 与业务关联）
- 时间相关性分析：告警/事件与业务指标异常的时间相关性
- 已知问题确认：是否有已知问题与当前异常相关

**L0 声明**：本步骤只读，不发起任何变更。

## 步骤 5：综合可靠性报告

**Agent 动作**：向 STAROps 发起综合评估查询，要求 AI 回顾 thread 历史中 Phase 1-4 结果并输出综合评估。

**工具调用**：

```
@应用-<服务名> 综合最近 <时间范围> 的业务指标基线、应用指标关联、依赖拓扑分析和告警事件，给出一份业务可靠性评估报告：包含整体健康判断、SLO 达标率评估、风险点、影响范围和行动建议
```

> 注意：本步骤依赖 Phase 1-4 的结果。确保在同一 thread 内执行，AI 会自动回顾历史上下文。

**输出要求**：
- 整体健康状态表（业务指标 / 应用健康 / 依赖稳定性 / 告警态势）
- SLO 达标率评估：是否达标 / 偏离幅度
- 风险点列表（按优先级排列，附带行动项和预期效果）
- 影响范围评估（受影响服务 / 用户 / 功能 / 时长）
- 建议行动项

**L0 声明**：本步骤只读，不发起任何变更。

## 输出与交付

SOP 跑完后客户能拿到：
- **业务可靠性报告**：业务指标基线 + 应用指标关联 + 依赖拓扑 + 告警事件 + 综合评估
- **SLO 达标率评估**：业务是否达标
- **风险点与行动项**：按优先级排列的改进建议

## 失败与回滚

L0 只读，无回滚需求。失败处置：

| 步骤 | 可能失败 | 处置 |
|---|---|---|
| 步骤 1 | 业务指标无数据（未接入 APM） | 检查服务是否已接入；Phase 1 和 Phase 2 可能合并 |
| 步骤 2 | 应用指标与业务指标无关联 | 检查埋点是否完整；手动补充已知关联 |
| 步骤 3 | 依赖拓扑不完整 | trace 采样率过低；prompt 中补充已知依赖关系 |
| 步骤 4 | 告警/事件数据缺失 | 检查告警系统是否接入 STAROps |
| 步骤 5 | 未引用前序结果 | 确认同一 thread；跨 thread 需重跑 |

需要人工介入：APM 数据源未接入、trace 采样率配置、告警系统对接。

## 召回 Routing

以下提问应路由到本 SOP Skill：
- "业务是否健康"
- "订单成功率怎么样"
- "登录服务可靠性评估"
- "核心 API 的 SLO 达标吗"
- "业务指标异常，帮我分析一下"
- "大促前业务健康确认"

以下提问不应路由：
- 告警触发后的被动排查 → `alert-rca-flow`
- 纯基础设施巡检 → `inspection-k8s-app-resource`
- 日志模式分析 → `log-insight-pattern`
